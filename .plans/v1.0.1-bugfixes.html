v1.0.1 Critical Bug Fixes

Context

After releasing v1.0.0 to the JetBrains Marketplace and testing on macOS, two critical bugs were discovered:

  1. Version switching breaks LSP — Changing the Nextflow language server version in settings causes all language server functionality to cease permanently (until IDE restart).
  2. DAG diagram not rendering — Users see raw mermaid text instead of a rendered diagram. Observed on macOS; works on the dev Ubuntu machine.

Both bugs are release-blocking for a usable v1.0.1. The version-switch bug is tackled first because it can be diagnosed and fixed entirely on the Linux dev machine, while the DAG bug requires macOS testing.

Phase 1: Diagnose version-switch LSP failure

The restart flow in NextflowLspConfigurationNotifier calls manager.stop() then manager.start() synchronously. The new server starts but likely never receives workspace initialization because warmUpProject() only runs during the one-time startup activity.

Add logging around the stop/start sequence in NextflowLspConfigurationNotifier.notifyChanged() — log before stop, after stop, before start, after start.
Check LSP4IJ's LanguageServerManager.stop() and start() API to determine if they are synchronous or async, and whether start() returns a future that can be chained.
Reproduce the bug: change version in settings, observe IDE logs to see whether the new server process actually starts, and whether it receives the LSP initialize handshake from LSP4IJ.
Verify hypothesis: after restart, check whether ensureWorkspaceInitialized() is ever called for the new server instance by adding logging to NextflowLspRuntime.

Phase 2: Fix version-switch restart

Make the LSP restart robust so the new server is fully operational after a version change.

Refactor NextflowLspConfigurationNotifier.notifyChanged() to properly sequence the restart: stop the server, wait for stop to complete (or use LSP4IJ's restart API if available), then start the new server.
After the new server starts, call NextflowLspRuntime.warmUpProject(project) to send workspace folders, synchronize open documents, and warm up symbols — mirroring what the startup activity does.
Clear stale state in NextflowLspRuntime on restart: the initializedServers and synchronizedDocuments WeakHashMaps hold references keyed by the old LanguageServer instance. While WeakHashMap will eventually GC these, consider adding an explicit clearState() method called before restart to avoid any window of confusion.
Remove temporary diagnostic logging added in Phase 1.

Phase 3: Test version-switch fix

Test version switching on Linux: change version, verify LSP reconnects, completions/diagnostics/hover all work with the new server version.
Test switching back to the original version to verify round-trip works.
Test changing Java home path (also triggers restart) to verify it works correctly too.

Phase 4: Diagnose DAG rendering failure on macOS

Determine exactly why the mermaid diagram doesn't render. The bundled mermaid.min.js (3.3 MB) is inlined into the HTML via loadHTML(). Three hypotheses to test:

Add temporary diagnostic logging to NextflowDagPreviewEditor: log whether JBCefApp.isSupported() is true, and log the size of the generated HTML string before passing it to loadHTML().
Build a dev/sandbox version and test on macOS. Check IDE logs (Help > Show Log) for the diagnostic output. Determine which path executes: JCEF browser or plain-text fallback.
If JCEF is supported but the diagram still doesn't render: test whether the 3.3 MB inline script exceeds loadHTML() / data-URI limits by trying a minimal mermaid snippet with the full bundled JS. Check the JCEF dev console (JBCefBrowser.openDevtools()) for JavaScript errors.

Phase 5: Fix DAG rendering

Apply the fix based on the diagnosis from Phase 4. Likely solutions ranked by probability:

If loadHTML() size limit is the issue: Switch from loadHTML() to writing a temp HTML file and loading it via browser.loadURL("file://..."). This avoids data-URI size limits and is how most JCEF-based previews in JetBrains plugins work. Store mermaid.min.js as a separate temp file referenced by a <script src="file://..."> tag.
If JCEF is unsupported on macOS: Add a user-visible warning when JCEF is unavailable, explaining that DAG preview requires JCEF. Consider providing instructions for enabling JCEF in JBR settings, or suggest using an IDE version that bundles it.
If mermaid.render() throws on macOS JCEF: Investigate the specific JS error. May need to adjust mermaid configuration, use a different mermaid version, or work around JCEF JavaScript engine differences.
Remove temporary diagnostic logging added in Phase 4.

Phase 6: Test DAG fix cross-platform

Test DAG preview on macOS with the fix applied — verify the diagram renders, node click navigation works, and refresh works.
Test DAG preview on Linux to verify no regression.
Test the JCEF-unavailable fallback path (if applicable) to ensure it degrades gracefully.

Phase 7: Release v1.0.1

Bump version in gradle.properties from 1.0.0 to 1.0.1.
Update changeNotes in build.gradle.kts to describe the bug fixes.
Build, sign, and publish to JetBrains Marketplace.
Install on macOS and verify both fixes work in the released plugin.

Notes

2026-05-15: Plan created. Key files involved:

  • NextflowLspConfigurationNotifier.kt — LSP stop/start on version change
  • NextflowLspRuntime.kt — workspace initialization and document sync state
  • NextflowStartupActivity.kt — one-time warm-up that doesn't re-run on restart
  • NextflowSettingsConfigurable.kt — settings UI, triggers restart
  • NextflowDagPreviewEditor.kt — DAG rendering, JCEF browser, mermaid.js inlining

Version-switch bug is tackled first (Phases 1-3) since it can be fully diagnosed and fixed on Linux. DAG rendering bug (Phases 4-6) requires macOS access for testing.

The bundled mermaid.min.js is 3.3 MB. Inlining this into loadHTML() creates a very large data URI which is the most likely cause of the macOS DAG failure. The version-switch bug is almost certainly caused by the new server never receiving workspace initialization after restart.
