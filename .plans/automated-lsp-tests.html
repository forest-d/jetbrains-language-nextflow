Automated Integration Tests for LSP Features

Context

The plugin has 69 manual test cases (PARITY_CHECKLIST.md) but only 32 automated tests. Key user-facing features — CodeLens "Preview DAG", hover documentation, completion, diagnostic filtering, DAG node navigation, and symbol categorization — have no automated coverage. Every code change requires tedious manual verification. This plan adds layered automated tests: pure unit tests (fast, no IDE), platform integration tests (BasePlatformTestCase with fixtures), and mock-server tests (Proxy-based LanguageServer mocks).

Phase 1: Pure Unit Tests

No IDE context needed. Fast, zero flakiness. Cover pure logic that currently has no tests.

SymbolCategory.from() — 15 tests in NextflowSymbolCategoryTest (process/workflow/function/record/enum/other, priority rules, case insensitivity)
displaySymbolName() — 7 tests in NextflowSymbolCategoryTest (prefix stripping for all types, unchanged when unprefixed, chained stripping)
dagNodeSymbolCandidates() — 14 tests in DagSymbolCandidatesTest (brackets, quotes, whitespace, keywords, compound labels, dedup, case filtering)
Diagnostic filtering — 7 tests in NextflowDiagnosticFilterTest (OFF/ERRORS/WARNINGS/PARANOID modes, null severity handling)
variableHover() — 8 tests in NextflowHoverSupportTest (assignment hover, take: hint, multi-line, HTML escaping, edge cases)

Phase 2: Platform Integration Tests

BasePlatformTestCase with fixture files. Tests the full contributor/provider pipeline.

paramHover() with schema — verify hover HTML for params.genome includes type, description, default, enum
paramHover() without schema — verify fallback to config-only default value
Completion: PROCESS.out. — myFixture.completeBasic() verifies emit names appear
Completion: params. — verify schema+config param names appear (also fixed bug: matchEntire→find in findConfigParamNames)
Completion: Channel. — verify factory method names appear
Completion: uppercase prefix in workflow — verify process names appear

Phase 3: Mock-Server Integration Tests

Expand the Proxy mock pattern from NextflowLspProtocolTest to test LSP request/response flows.

Build reusable MockLanguageServer returning canned codeLens, resolveCodeLens, executeCommand responses
Test CodeLens retrieval — mock returns Preview DAG lens, verify findDagCommand selects closest to caret (+ ignores non-DAG lenses, falls back on resolve failure)
Test CodeLens retry — mock returns empty then lens on attempt N, verify retry logic (+ fails after 8 attempts)
Test DAG command execution — 14 DagPreviewResult tests (toMermaid, errorMessage) + 4 executeDagCommand tests (command mapping, argument passthrough)
Test findLocalSymbol — 5 tests: process/workflow/def in same and different files, correct line/character, null for missing symbols

Phase 4: Regression Guard Tests

Targeted tests for known fragile interactions.

ensureDocumentSynchronized(force=false) + documentSymbol: verify no didOpen for editor-managed files even when followed by symbol requests
Diagnostic mode switching: verify clearDiagnostics() re-publishes empty for all known URIs

Notes

Existing test patterns: pure @Test (kotlin.test) for unit tests, BasePlatformTestCase (JUnit 3 naming) for platform tests, java.lang.reflect.Proxy for mock LanguageServer. Fixture project at src/test/resources/fixtures/parity/ with main.nf, modules/, nextflow.config, nextflow_schema.json.
