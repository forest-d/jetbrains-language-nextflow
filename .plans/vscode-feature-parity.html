VS Code Feature Parity Testing & Gap Closure

Context

The JetBrains Nextflow plugin (built in the pycharm-nextflow-plugin plan) needs systematic verification against the VS Code Nextflow extension (Marketplace) to confirm feature parity and close remaining gaps.

Scope

  • In scope: all language-editing, navigation, visualization, and configuration features from the VS Code extension.
  • Out of scope: AI/Copilot features, Seqera Cloud integration, telemetry, type conversion commands (deferred).

Phase 1: Test Fixtures & Verification Infrastructure

Create a comprehensive set of Nextflow sample files that exercise every language feature, plus a structured manual test checklist and any automated test additions.

Create a test fixture project at src/test/resources/fixtures/parity/ with a nextflow.config that defines params, profiles, and process directives
Create main.nf fixture with: multiple processes (with typed and untyped inputs/outputs), workflows calling processes, channel operations (map, filter, collect, mix), function definitions, and conditional logic
Create modules/sample_module.nf fixture with a process and workflow to test cross-file navigation and extended completion
Create nextflow_schema.json fixture to test parameter schema hover/completion support
Create a PARITY_CHECKLIST.md in .plans/ with a structured manual test script: for each feature, list the exact steps to perform, expected result, and a pass/fail column

Phase 2: Verify Existing Feature Parity

Run through each LSP-backed and UI feature against the test fixtures, documenting results. Fix any regressions or gaps found during verification.

Syntax highlighting: open each fixture file type (.nf, .nf.test, nextflow.config) and visually verify keyword, string, comment, and operator highlighting matches VS Code output
Diagnostics: introduce deliberate errors (typos, missing imports, type mismatches) and verify red/yellow squiggles appear; toggle errorReportingMode between all levels and confirm behavior changes
Hover: hover over process names, function calls, channel variables, and params; verify tooltip shows definition and any Javadoc; compare content with VS Code hover output
Code completion: trigger completion inside a workflow body for process names, channel methods, and params; verify extended completion surfaces symbols from modules/sample_module.nf when enabled
Code navigation: Ctrl+click on a process call to verify go-to-definition; use Find Usages on a process name to verify find-references; test cross-file navigation to modules/
Symbol renaming: rename a process and verify all call sites update; rename a channel variable and verify downstream references update
Formatting: reformat main.nf with each formatter option toggled independently (harshil, mahesh, sort) and verify output matches VS Code formatting for the same options
DAG preview: open DAG for a workflow with multiple processes; verify Mermaid renders correctly, node click navigates to source, and the diagram updates on file edit
Project view: verify the Nextflow tool window lists all processes, workflows, and functions from all fixture files; verify double-click navigates to definition
Status bar: verify the widget shows correct states during startup, while idle, and after stopping the server
Settings persistence: change each setting, restart IDE, verify values are preserved and sent to LSP server
Document results in the parity checklist with pass/fail and notes on any discrepancies

Phase 3: Code Snippets (Live Templates)

Add IntelliJ Live Templates for common Nextflow constructs, matching or exceeding the VS Code extension's built-in snippets.

Research VS Code extension's snippet definitions (check the language server or extension source for snippet content)
Create a Live Template group "Nextflow" registered via plugin.xml
Add process snippet: expands to a full process block with input/output/script sections and cursor placeholders
Add workflow snippet: expands to a workflow block with take/main/emit sections
Add channel snippet: common channel factory patterns (Channel.of, Channel.fromPath, Channel.fromFilePairs)
Add additional snippets for: function definition, conditional process execution (when clause), publishDir directive, container directive
Set template applicability context to Nextflow file types only
Verify snippets expand correctly in the IDE with tab stops and placeholders working

Phase 4: Missing Settings & Configuration

Add the typeChecking setting and ensure all settings are correctly propagated to the language server.

Research the nextflow.typeChecking setting: determine what values the language server accepts (boolean? enum?) by checking the LS source or VS Code extension package.json
Add typeChecking field to NextflowSettings.State with appropriate type and default
Add the type checking UI control to NextflowSettingsConfigurable
Wire the new setting into toLspSettings() and toFlatLspSettings()
Add a test to NextflowSettingsTest verifying the new setting serializes correctly
Verify in IDE: enable type checking, introduce a type error, confirm the LSP reports it

Phase 5: Project View Enhancement

Add a hierarchical call-tree mode to the Nextflow tool window, matching the VS Code extension's tree view that organizes processes and workflows by call hierarchy.

Research how the VS Code extension builds its call hierarchy tree (likely using the LSP callHierarchy requests or walking code lenses)
Add a toggle button to the project view toolbar to switch between flat (current) and tree (hierarchy) modes
Implement the tree mode: for each workflow, show the processes and sub-workflows it calls as children
Preserve navigation behavior (double-click opens definition) in tree mode
Verify the tree view reflects the actual call structure of the test fixtures and matches VS Code's tree view

Phase 6: Final Parity Audit & Regression Tests

Perform a side-by-side comparison of every feature between the VS Code extension and our JetBrains plugin. Add automated regression tests where feasible.

Open the same test project in both VS Code (with Nextflow extension) and JetBrains IDE (with our plugin) side by side
Walk through every item in the parity checklist; screenshot any remaining discrepancies
Add automated tests for snippet expansion (verify template XML parses and context is set correctly)
Add automated tests for settings completeness (verify all VS Code settings have JetBrains equivalents in toFlatLspSettings())
Update the feature matrix in this plan with final pass/fail results
Document any intentional divergences (features we do better or differently) in the Notes section

Notes

2026-05-11: Plan created. Key decisions:

  • Type conversion commands deferred — the VS Code extension offers "Convert to static types" via Command Palette, but this is a lower-priority feature that can be added later.
  • Seqera/AI/telemetry excluded — these are intentionally out of scope for this plugin.
  • Test fixture approach — a single comprehensive Nextflow project that exercises all features is more useful than many tiny isolated files, because it also tests cross-file navigation and extended completion.
  • Live Templates over LSP snippets — IntelliJ's Live Template system is richer than VS Code snippets (supports Groovy expressions, surround templates, etc.) and is the idiomatic JetBrains approach. Users can also customize them in Settings → Editor → Live Templates.
  • Project view hierarchy — the VS Code extension's tree view is a differentiating feature; it may require call hierarchy LSP support or walking the DAG structure. Research needed in Phase 5.
