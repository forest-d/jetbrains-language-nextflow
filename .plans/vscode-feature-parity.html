Context

Scope

  • In scope: all language-editing, navigation, visualization, and configuration features from the VS Code extension.
  • Out of scope: AI/Copilot features, Seqera Cloud integration, telemetry, type conversion commands (deferred).

Bug: Diagnostics Only Fire on Save (not real-time)

Symptom: Editing a file (e.g., renaming FASTQC to FASTQX) does not produce red squiggles until the user presses Ctrl+S. VS Code shows diagnostics in real-time as the user types.

Investigation:

  • The Nextflow LS reports TextDocumentSyncKind.Incremental (value 2) and handles textDocument/didChange with a delayed-update mechanism. Other editors (Neovim, Helix) get real-time diagnostics from the same server.
  • LSP4IJ creates a DocumentContentSynchronizer that listens for IntelliJ document changes and sends textDocument/didChange. It debounces via PsiDocumentManager.performForCommittedDocument().
  • Attempted fix 1: Added NextflowLanguageClient with handleServerStatusChanged(started) that sends workspace/didChangeWorkspaceFolders and didChangeConfiguration. Result: diagnostics appeared on initial load and on save, but not on edit.
  • Attempted fix 2: After workspace init, manually re-sent textDocument/didOpen for all open Nextflow files. Result: no improvement; the manual didOpen likely conflicted with LSP4IJ’s internal version tracking, preventing its didChange from reaching the server correctly.
  • Intermediate state: Stripped NextflowLanguageClient to only override createSettings(), relying entirely on LSP4IJ for document lifecycle. Diagnostics appear on initial load and on save, not on edit.
  • Root cause: LSP4IJ 0.19.3 routes editor changes through PsiDocumentManager.performForCommittedDocument(). For the TextMate-backed Nextflow file type this can leave textDocument/didChange unsent until save, while the Nextflow LS itself handles didChange correctly.
  • Fix: Added NextflowRealtimeDiagnostics, a debounced JetBrains document listener registered at project startup. It sends full-document textDocument/didChange notifications for .nf, .nf.test, and nextflow.config edits using the current unsaved editor text and a null version so it does not fight LSP4IJ’s internal version counter.
  • Status: Fixed and manually verified. Checklist item 2.2 now passes: after editing FASTQC to FASTQX, diagnostics appear without saving.

Bug: Error Reporting Mode Toggle Has No Effect

Symptom: Changing Settings → Nextflow → Error Reporting from “Warnings” to “Off” and clicking Apply does not remove existing squiggles.

Root cause: The Nextflow LS has an upstream bug where didChangeConfiguration can never apply settings at runtime. LSP4J’s standard Gson deserialization converts the Object-typed settings field into a LinkedTreeMap, but the LS’s JsonUtils.getObjectPath() checks if (!(json instanceof JsonObject)) return null. Since LinkedTreeMap instanceof JsonObject is always false, the settings are silently dropped regardless of format (flat or nested). This was confirmed by decompiling the actual LS JAR classes.

An initial fix attempt (changing all settings emission paths from flat to nested format) was reverted after manual testing showed it had no effect — the upstream type mismatch made both formats fail.

Fix: Client-side diagnostic filtering, bypassing the broken server-side settings entirely. Three coordinated changes:

  • NextflowLanguageClient.publishDiagnostics() — overrides the LSP notification handler to filter incoming diagnostics based on the current ErrorReportingMode (OFF: empty list, ERRORS: severity filter, WARNINGS/PARANOID: pass-through). Tracks all published URIs for later clearing.
  • NextflowLspConfigurationNotifier.refreshDiagnostics() — when the error reporting mode changes, clears all existing squiggles via client.clearDiagnostics() (publishes empty lists for all tracked URIs), then re-syncs open Nextflow files to trigger fresh diagnostics from the LS (which pass through the updated filter).
  • NextflowSettingsConfigurable.apply() — tracks the old error mode and passes an errorModeChanged flag to the notifier.

Status: Fixed and manually verified. Switching to “Off” immediately clears all squiggles.

Bug: Hover (textDocument/hover) Not Working

Symptom: Hovering over process names, function calls, channel variables, or params in Nextflow files produces no tooltip. Ctrl+Q (Quick Documentation) also shows nothing. The Nextflow LS reports hoverProvider: true in its capabilities, and hover works correctly in VS Code with the same server.

Investigation:

  • LSP4IJ implements hover via the platform.backend.documentation.targetProvider extension point (LSPDocumentationTargetProvider). This provider takes a PsiFile + offset and sends textDocument/hover to the language server. It is registered with order="first" in LSP4IJ’s plugin.xml.
  • Diagnostics work (server-pushed via publishDiagnostics), proving the language server is running and connected. But hover is client-initiated and uses a completely different code path.
  • LSP4IJ also registers a LSPSemanticTokensStructurelessFileViewProviderFactory at project startup for file types that don’t have their own PSI structure (e.g., TextMate-backed files). This factory creates FileViewProviders that produce LSPSemanticTokenPsiElements, which IntelliJ’s documentation framework uses to trigger hover.
  • Root cause: LSP4IJ’s LSPSemanticTokensStructurelessFileViewProviderRegistrar.registerAbstractFileTypes() only processes file types that are instanceof AbstractFileType (IntelliJ’s class for user-defined file types). Our plugin’s NextflowFileType, NextflowTestFileType, and NextflowConfigFileType implement FileType directly (with TextMateBackedFileType), so they are skipped by the registrar. Without the semantic token FileViewProvider, Nextflow files get plain-text PSI (PsiPlainTextFile) with no meaningful elements, and IntelliJ’s documentation framework does not trigger hover.

Fix: Manually register LSPSemanticTokensStructurelessFileViewProviderFactory for all three Nextflow file types in NextflowStartupActivity via FileTypeFileViewProviders.INSTANCE.addExplicitExtension(). This is done once (guarded by a static flag) before the diagnostics bridge is installed, ensuring the semantic token FileViewProvider is active when files are opened.

Status: Fixed. Needs manual verification.
