Context

Scope

  • In scope: all language-editing, navigation, visualization, and configuration features from the VS Code extension.
  • Out of scope: AI/Copilot features, Seqera Cloud integration, telemetry, type conversion commands (deferred).

Bug: Diagnostics Only Fire on Save (not real-time)

Symptom: Editing a file (e.g., renaming FASTQC to FASTQX) does not produce red squiggles until the user presses Ctrl+S. VS Code shows diagnostics in real-time as the user types.

Investigation:

  • The Nextflow LS reports TextDocumentSyncKind.Incremental (value 2) and handles textDocument/didChange with a delayed-update mechanism. Other editors (Neovim, Helix) get real-time diagnostics from the same server.
  • LSP4IJ creates a DocumentContentSynchronizer that listens for IntelliJ document changes and sends textDocument/didChange. It debounces via PsiDocumentManager.performForCommittedDocument().
  • Attempted fix 1: Added NextflowLanguageClient with handleServerStatusChanged(started) that sends workspace/didChangeWorkspaceFolders and didChangeConfiguration. Result: diagnostics appeared on initial load and on save, but not on edit.
  • Attempted fix 2: After workspace init, manually re-sent textDocument/didOpen for all open Nextflow files. Result: no improvement; the manual didOpen likely conflicted with LSP4IJ’s internal version tracking, preventing its didChange from reaching the server correctly.
  • Intermediate state: Stripped NextflowLanguageClient to only override createSettings(), relying entirely on LSP4IJ for document lifecycle. Diagnostics appear on initial load and on save, not on edit.
  • Root cause: LSP4IJ 0.19.3 routes editor changes through PsiDocumentManager.performForCommittedDocument(). For the TextMate-backed Nextflow file type this can leave textDocument/didChange unsent until save, while the Nextflow LS itself handles didChange correctly.
  • Fix: Added NextflowRealtimeDiagnostics, a debounced JetBrains document listener registered at project startup. It sends full-document textDocument/didChange notifications for .nf, .nf.test, and nextflow.config edits using the current unsaved editor text and a null version so it does not fight LSP4IJ’s internal version counter.
  • Status: Fixed and manually verified. Checklist item 2.2 now passes: after editing FASTQC to FASTQX, diagnostics appear without saving.

Bug: Error Reporting Mode Toggle Has No Effect

Symptom: Changing Settings → Nextflow → Error Reporting from “Warnings” to “Off” and clicking Apply does not remove existing squiggles.

Root cause: The Nextflow LS has an upstream bug where didChangeConfiguration can never apply settings at runtime. LSP4J’s standard Gson deserialization converts the Object-typed settings field into a LinkedTreeMap, but the LS’s JsonUtils.getObjectPath() checks if (!(json instanceof JsonObject)) return null. Since LinkedTreeMap instanceof JsonObject is always false, the settings are silently dropped regardless of format (flat or nested). This was confirmed by decompiling the actual LS JAR classes.

An initial fix attempt (changing all settings emission paths from flat to nested format) was reverted after manual testing showed it had no effect — the upstream type mismatch made both formats fail.

Fix: Client-side diagnostic filtering, bypassing the broken server-side settings entirely. Three coordinated changes:

  • NextflowLanguageClient.publishDiagnostics() — overrides the LSP notification handler to filter incoming diagnostics based on the current ErrorReportingMode (OFF: empty list, ERRORS: severity filter, WARNINGS/PARANOID: pass-through). Tracks all published URIs for later clearing.
  • NextflowLspConfigurationNotifier.refreshDiagnostics() — when the error reporting mode changes, clears all existing squiggles via client.clearDiagnostics() (publishes empty lists for all tracked URIs), then re-syncs open Nextflow files to trigger fresh diagnostics from the LS (which pass through the updated filter).
  • NextflowSettingsConfigurable.apply() — tracks the old error mode and passes an errorModeChanged flag to the notifier.

Status: Fixed and manually verified. Switching to “Off” immediately clears all squiggles.

Bug: Hover (textDocument/hover) Not Working

Symptom: Hovering over process names, function calls, channel variables, or params in Nextflow files produces no tooltip. Ctrl+Q (Quick Documentation) also shows nothing. The Nextflow LS reports hoverProvider: true in its capabilities, and hover works correctly in VS Code with the same server.

Investigation:

  • LSP4IJ implements hover via the platform.backend.documentation.targetProvider extension point (LSPDocumentationTargetProvider). This provider takes a PsiFile + offset and sends textDocument/hover to the language server. It is registered with order="first" in LSP4IJ’s plugin.xml.
  • Diagnostics work (server-pushed via publishDiagnostics), proving the language server is running and connected. But hover is client-initiated and uses a completely different code path.
  • LSP4IJ also registers a LSPSemanticTokensStructurelessFileViewProviderFactory at project startup for file types that don’t have their own PSI structure (e.g., TextMate-backed files). This factory creates FileViewProviders that produce LSPSemanticTokenPsiElements, which IntelliJ’s documentation framework uses to trigger hover.
  • Root cause (initial hypothesis — wrong): Initially thought LSP4IJ’s LSPSemanticTokensStructurelessFileViewProviderRegistrar was skipping our file types because they aren’t AbstractFileType. Added manual FileViewProvider registration in NextflowStartupActivity. This did not fix hover.
  • Root cause (actual): The TextMate plugin claims .nf, .nf.test, and .config extensions via the bundled package.json, overriding our custom NextflowFileType registrations. At runtime, files get TextMateFileType instead of NextflowFileType. LSP4IJ’s fileTypeMapping was registered for "Nextflow", so LanguageServersRegistry.isFileSupported(psiFile) returned false and canExecuteLSPFeature returned NOT. This blocked hover and all other client-initiated LSP features. Diagnostics still worked because the server is started explicitly by our code and diagnostics are server-pushed.

Fix: Replaced fileTypeMapping with fileNamePatternMapping in plugin.xml. Instead of matching by file type (unreliable due to the TextMate conflict), we now match by filename pattern: *.nf, *.nf.test, and nextflow.config. This bypasses file type resolution entirely and makes isFileSupported return true.

Status: Fixed and manually verified.

Bug: Code Completion for Named Process Outputs Returned No Items

Symptom: Typing FASTQC.out. in a workflow body shows a large list of irrelevant items (IntelliJ word completion) instead of the process’s outputs. No LSP-sourced completion items appear at all.

Investigation:

  • LSP4IJ’s LSPCompletionContributor runs (registered with language="any", order="first, before wordCompletion"). The temporary NextflowCompletionFeature.isSupported() returned true, confirming the server reports completionProvider in its capabilities.
  • However, addLookupItem() is never called — no items reach the client-side feature layer.
  • A direct textDocument/completion request sent to the server at the trailing-dot cursor (bypassing LSP4IJ entirely) also returns 0 items. This confirms the Nextflow Language Server itself is not providing named output completions for this cursor state.
  • The same direct debug request at line 1, char 0 returns top-level declaration snippets, proving the server’s completion endpoint is alive.
  • The bug reproduces both in the main IDE and in the sandbox with the fixtures opened as a standalone project, ruling out workspace-root misconfiguration.
  • Hover and diagnostics work for the same file, so the server IS running and connected. These use different code paths (server-pushed diagnostics, textDocument/hover).

Root cause: The upstream Nextflow Language Server does not return completion items at a trailing-dot PROCESS.out. position. Its script completion provider requires the cursor to resolve to an AST node such as a PropertyExpression; at this incomplete member-access state the request returns 0 items even though process outputs are declared and other LSP features work.

Fix: Added a narrow JetBrains-side fallback contributor for this exact shape. NextflowOutputCompletionContributor detects PROCESS.out. / PROCESS.out.prefix, extracts named outputs from the local process’s output: block by reading emit: names, and contributes those items before word completion. This restores FASTQC.out. suggestions without changing general LSP completion behavior.

  • NextflowOutputCompletionSupport.kt — detects output access at the completion offset and extracts distinct emit: names from the matching local process output block.
  • NextflowOutputCompletionContributor.kt — contributes matching output names and stops lower-priority word completion for this context.
  • NextflowOutputCompletionSupportTest.kt — covers empty-prefix FASTQC.out., prefix-filtered FASTQC.out.re, output extraction, and non-output contexts.
  • Removed the temporary debug NextflowCompletionDebugContributor and NextflowCompletionFeature wiring.

Status: Fixed and manually verified. FASTQC.out. now offers reports and zips.

Bug: Code Navigation Does Not Work

Manual test results: All section 5 navigation checks fail. Ctrl+click does not navigate for local process calls, included process calls, function calls, or include paths. Find Usages and Outline/Structure are also not working.

Investigation update: Removed the broken experimental NextflowGotoDeclarationHandler. It navigated to the top of the file and interfered with previously working hover behavior.

Root cause hypothesis: Manual testing showed hover and ctrl-click work after pressing the Nextflow tool window Refresh button. That refresh path initializes the workspace, synchronizes project Nextflow documents, and requests document symbols, which appears to prime the Nextflow LS semantic state needed by LSP4IJ navigation.

Fix: Added automatic startup warm-up via NextflowLspRuntime.warmUpProject(). After the language server is available, startup now initializes the workspace, synchronizes all project Nextflow files, and requests document symbols for those files. This mirrors the relevant refresh behavior without requiring the user to open the tool window or press Refresh manually.

Status: Fix implemented; needs fresh IDE load retest. Find Usages and Outline/Structure remain open.

Bug: DAG Preview Crash on File Edit

Symptom: IncorrectOperationException: File is not writable when typing in a .nf file that has a DAG preview open. The auto-refresh timer calls NextflowDagPreviewFile.updateMermaid() which called LightVirtualFile.setContent(), but the file overrode isWritable() to return false.

Fix: Removed the setContent() call from updateMermaid() — the mermaid field is updated directly and render() uses that field, so the VFS content sync was unnecessary.

Status: Fixed.
