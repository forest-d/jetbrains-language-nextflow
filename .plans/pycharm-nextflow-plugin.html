PyCharm Nextflow Plugin

Context

Build a PyCharm/IntelliJ plugin that provides full Nextflow language support, achieving feature parity with the existing VS Code Nextflow extension.

Architecture: Kotlin + LSP4IJ

The plugin is pure Kotlin, using IntelliJ Platform SDK and LSP4IJ.

  • The Nextflow Language Server already exists as a standalone Java application — no need to rewrite parsing/analysis.
  • IntelliJ plugins are JVM-native; Kotlin is the standard language with first-class SDK support.
  • LSP4IJ provides mature LSP client support across all IntelliJ flavors (PyCharm, IDEA CE/Ultimate, etc.).
  • IntelliJ supports TextMate grammar bundles natively — the VS Code grammars can be reused.

Reference: VS Code Extension Feature Set

  • Syntax highlighting (TextMate grammars for .nf, .nf.test, .config)
  • LSP features: completion, diagnostics, hover, go-to-definition, references, rename, formatting
  • DAG preview (Mermaid-based workflow visualization)
  • Project view (processes, workflows, tests tree)
  • Seqera Cloud integration (auth, workspaces, pipelines, datasets, runs)
  • Language server version management & auto-download
  • Java runtime detection
  • Telemetry (opt-in, PostHog)

Phase 1: Project Scaffold & Build System

Starting from a fresh IntelliJ IDEA Kotlin project. Configure it as an IntelliJ Platform plugin using the IntelliJ Platform Gradle Plugin.

Add the org.jetbrains.intellij.platform plugin (IntelliJ Platform Gradle Plugin 2.x) to the existing build.gradle.kts
Configure Gradle to target IntelliJ 2024.3+ (broad compatibility) with Kotlin 2.0+ and Java 21
Add IntelliJ Platform dependencies (intellijPlatform block) and plugin repositories to settings.gradle.kts
Add version, group, and platform version properties to gradle.properties
Create src/main/resources/META-INF/plugin.xml with plugin ID, name, vendor, description, and dependency on com.intellij.modules.platform
Create a minimal plugin startup activity class (NextflowStartupActivity.kt) that logs activation
Update .gitignore for build artifacts (build/, .gradle/)
Verify the plugin builds and loads in a sandbox IDE via ./gradlew runIde

Phase 2: File Type Registration & Syntax Highlighting

Register Nextflow file types and provide syntax highlighting by bundling TextMate grammars from the VS Code extension.

Register file types: Nextflow Script (.nf), Nextflow Test (.nf.test), Nextflow Config (.config) in plugin.xml
Create file type icons (16x16, 13x13) for .nf, .nf.test, and .config files
Copy TextMate grammar files from VS Code extension into src/main/resources/textmate/: nextflow.tmLanguage.json, nextflow-config.tmLanguage.json, groovy.tmLanguage.json
Register TextMate bundles in plugin.xml using the com.intellij.textmate.bundleProvider extension point
Create a TextMate bundle provider class that maps scopes to file types
Configure language-specific editor behavior: bracket matching, auto-close pairs, comment toggling (port from language-configuration.json)
Verify syntax highlighting works for .nf, .nf.test, and .config files in sandbox IDE

Phase 3: Language Server Management

Implement the infrastructure to download, cache, and launch the Nextflow Language Server JAR — mirroring what the VS Code extension does in fetchLanguageServer.ts.

Add LSP4IJ as a plugin dependency in build.gradle.kts and plugin.xml
Create NextflowLanguageServerFactory extending LanguageServerFactory from LSP4IJ
Implement Java runtime detection: check plugin setting → JAVA_HOME env → PATH; require Java 17+
Implement language server JAR download from GitHub releases (https://github.com/nextflow-io/language-server/releases) with version selection
Implement local JAR caching at ~/.nextflow/lsp/v{version}/ (same path as VS Code extension for shared caching)
Implement fallback behavior: use cached version if download fails (offline support)
Create a startup activity that checks for/downloads the language server on plugin load
Configure LSP4IJ server definition: stdio transport, document selectors for .nf/.nf.test/.config, initialization options
Verify language server starts and connects — check that hover/completion basics work in sandbox IDE

Phase 4: LSP Feature Wiring & Settings

Wire up all LSP-powered features through LSP4IJ and expose plugin settings matching the VS Code extension's configuration.

Verify code completion works and configure maxItems setting (default: 100)
Verify diagnostics/error reporting and configure error reporting mode (off, errors, warnings, paranoid)
Verify go-to-definition, find references, and rename work through LSP4IJ
Verify hover information displays correctly
Verify document formatting works and configure formatting settings (harshilAlignment, maheshForm, sortDeclarations)
Create a settings page (Configurable) under Languages & Frameworks → Nextflow with all settings
Implement settings persistence using PropertiesComponent or PersistentStateComponent
Wire settings changes to LSP workspace/didChangeConfiguration notifications
Add a "files.exclude" setting to control which directories the language server skips (default: .git, .nf-test, work)
Add a "languageVersion" setting to select language server version (26.04, 25.10, 25.04, 24.10)

Phase 5: Project Tool Window

Build a tool window that shows the project's Nextflow structure: processes, workflows, functions, and tests — equivalent to the VS Code extension's Project View webview.

Create a tool window factory (NextflowProjectToolWindowFactory) registered in plugin.xml
Implement a tree model that queries the language server for workspace symbols (textDocument/documentSymbol or workspace/symbol)
Display processes, workflows, functions, records, and enums in a hierarchical tree
Add click-to-navigate: clicking a tree node opens the file at the symbol's location
Add tree node icons for each symbol type (process, workflow, function, etc.)
Implement auto-refresh when files change (listen to VFS events or LSP notifications)
Verify the tool window works correctly in sandbox IDE

Phase 6: DAG Preview

Implement a workflow DAG (directed acyclic graph) visualization, equivalent to the VS Code extension's Mermaid-based preview.

Research how the VS Code extension generates DAG data (check if the language server provides DAG info or if the extension builds it)
Create a DAG preview action accessible from the editor gutter or context menu
Implement DAG rendering using JCEF (embedded Chromium) with Mermaid.js — or a native Swing graph if JCEF is unavailable
Display the DAG in an editor tab or split view
Add click-to-navigate from DAG nodes back to source code
Implement auto-refresh when the workflow definition changes

Phase 7: Seqera Cloud Integration

Implement Seqera Cloud authentication and workspace management — equivalent to the VS Code extension's Seqera Cloud webview panel.

Implement OAuth authentication flow with Seqera Cloud (Auth0 provider at cloud.seqera.io)
Store authentication tokens securely using IntelliJ's CredentialStore / PasswordSafe API
Implement JWT token validation and refresh logic
Create a Seqera Cloud tool window with login/workspace UI
Implement workspace selector (organization + workspace dropdown)
Implement pipeline browser: list, view details, launch pipelines
Implement run history viewer
Implement dataset browser
Implement compute environment viewer
Add "Add Pipeline" action to deploy a pipeline to Seqera Cloud from the IDE

Phase 8: Testing, Polish & Distribution

Harden the plugin with tests, polish UX, and prepare for JetBrains Marketplace distribution.

Write unit tests for language server management (download, caching, version selection)
Write unit tests for Java runtime detection
Write unit tests for settings persistence and change notification
Write integration tests using IntelliJ's test framework (LightPlatformCodeInsightTestCase or similar)
Add error notifications for common failures: Java not found, language server download failed, server crashed
Add a "Nextflow" entry to the status bar showing language server status (starting, running, error)
Create plugin icon and screenshots for Marketplace listing
Write plugin description and changelog for Marketplace
Run Plugin Verifier (./gradlew verifyPlugin) against target IDE versions
Publish to JetBrains Marketplace (or prepare for private distribution)

Notes

2026-05-10: Plan created. Key decisions:

  • Kotlin + IntelliJ Platform SDK — standard JetBrains plugin architecture.
  • LSP4IJ chosen over JetBrains' native LSP API for broader IDE/version compatibility (works with PyCharm Community, all IntelliJ flavors, 2024.3+). JetBrains' native LSP API requires 2025.2+ and was initially limited to Ultimate editions.
  • Reuse TextMate grammars from VS Code extension rather than writing a custom lexer/parser — IntelliJ supports TextMate bundles natively and this avoids duplicating effort.
  • Share the JAR cache at ~/.nextflow/lsp/ with the VS Code extension so users who have both IDEs don't download the server twice.
  • Seqera Cloud integration is Phase 7 — it's valuable but not core to the language editing experience. Phases 1-4 deliver the essential editing features.
  • Starting point is a fresh IntelliJ IDEA Kotlin project — IDE generates Gradle wrapper, settings.gradle.kts, and basic project structure.

2026-05-10: Phase 1 complete. Implementation notes:

  • IntelliJ Platform Gradle Plugin 2.16.0 requires Gradle 9.0+ (not 8.x). Updated wrapper accordingly.
  • instrumentationTools() was removed in plugin 2.16.0 — no longer needed in the dependencies block.
  • Repository config moved from settings.gradle.kts to build.gradle.kts — the intellijPlatform { defaultRepositories() } DSL inside dependencyResolutionManagement didn't resolve with the settings plugin alone.
  • Startup activity uses ProjectActivity (coroutine-based) — the modern replacement for the deprecated StartupActivity.
  • JDK 21 installed locally at ~/.local/jdk (Adoptium Temurin 21.0.11).
  • Plugin builds to build/distributions/jetbrains-language-nextflow-0.1.0.zip.

2026-05-10: Phase 2 complete. Implementation notes:

  • TextMate bundle uses VS Code-style package.json format — IntelliJ's TextMate plugin reads it natively including language-configuration.json for brackets, comments, auto-close.
  • Three grammars bundled: nextflow.tmLanguage.json (scope: source.nextflow), nextflow-config.tmLanguage.json (scope: source.nextflow-config), groovy.tmLanguage.json (scope: source.nextflow-groovy, referenced by nextflow grammar).
  • Bundle provider — NextflowTextMateBundleProvider implements TextMateBundleProvider from org.jetbrains.plugins.textmate.api. Requires bundledPlugin("org.jetbrains.plugins.textmate") in Gradle deps.
  • File types registered with TextMateBackedFileType marker interface for custom icons. Three SVG icons: green "NF" for scripts, blue "CFG" for config, amber "TST" for tests.
  • No separate Commenter/BraceMatcher needed — the TextMate plugin provides these automatically from language-configuration.json.
  • .config association is only for files named nextflow.config (via fileNames) to avoid conflicting with generic .config files.

2026-05-10: Phase 3 complete. Implementation notes:

  • LSP4IJ 0.19.3 used as Marketplace plugin dependency (com.redhat.devtools.lsp4ij). Uses fileTypeMapping extension point with languageId attributes matching what the LS expects (nextflow and nextflow-config).
  • Java detection (JavaFinder) checks JAVA_HOME then PATH, validates version ≥ 17 by parsing java -version output.
  • Server download (LanguageServerDownloader) queries GitHub releases API for latest patch of version prefix (default v26.04), downloads language-server-all.jar, caches at ~/.nextflow/lsp/v26.04/v26.04.0.jar. Falls back to cached JAR if offline.
  • Connection provider (NextflowLanguageServer) extends ProcessStreamConnectionProvider, launches java -jar <server.jar> via stdio. Sends initialization options with files.exclude list.
  • Startup activity pre-downloads the LS JAR in background on project open so it's ready when the first .nf file is opened.
  • Latest LS release: v26.04.0 (language-server-all.jar, ~15MB).
